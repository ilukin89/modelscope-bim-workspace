# ModelScope BIM Workspace - Backend Implementation Design

## Purpose

This document turns the merged backend/auth/issue persistence scope into a
concrete Supabase implementation design.

It is documentation and design only. It does not add packages, Supabase client
code, environment variables, migrations, authentication UI, backend runtime
code, React component changes, renderer changes, AI Review behavior changes,
Drawing Triage behavior changes, or issue lifecycle runtime changes.

The goal is to give a later implementation PR enough detail to add Supabase
dependencies, environment configuration, migrations, seeds, and integration
code safely.

## Product Boundary

The first backend slice persists only the existing Model Review issue workflow:

```text
seeded demo user
-> project membership
-> AI scan run
-> AI finding
-> user decision
-> created issue
-> issue status history
-> review history event
```

Product rules:

- Do not collapse `ai_findings` and `model_review_issues` into one table.
- Scan completion must not create issues.
- Browser code must not generate permanent issue IDs.
- Do not store the frontend `sourceIssue` embedded object as a backend shape.
- Preserve full source lineage from scan run to finding to decision to issue to
  status history to review history.
- This is a seeded demo slice, not a full product backend.

## Why Supabase

Supabase is appropriate for this portfolio and demo phase because it provides
the smallest credible path from a frontend prototype to a durable, auditable
backend:

- Postgres gives a clear relational model for scan runs, findings, decisions,
  issues, and append-only history.
- Row Level Security maps directly to the project membership boundary required
  by the 011 spec.
- Supabase Auth provides a real authenticated session without building custom
  credential storage.
- RPC functions can run multi-table writes in a single database transaction.
- Seed scripts can create deterministic demo users, projects, scan runs, and
  findings without introducing public signup or production onboarding.

Alternatives considered:

| Option | Reason not chosen for this phase |
| --- | --- |
| Browser-only local state | Already exists, but cannot survive refreshes or provide user attribution. |
| Custom Node/Express backend | Adds more infrastructure than this seeded demo slice needs. |
| Firebase/Firestore | Less natural for relational source lineage and append-only workflow history. |
| SQLite/local file persistence | Useful for local demos, but does not exercise project membership or browser auth. |

## Auth and Session Model

### Options Evaluated

| Option | Fit | Risks |
| --- | --- | --- |
| Supabase anonymous sign-in | Simple browser session with no password UI. | Identity can become device/session shaped, harder to keep deterministic across the portfolio demo, and less clear for seeded project memberships. |
| Seeded email/password demo user | Deterministic Auth user, easy project membership seeding, clear attribution, and no public account creation. | Requires a controlled demo credential or pre-authenticated demo environment in a later implementation. |
| `service_role` access | Useful for seeds and admin scripts only. | Catastrophic if exposed to the browser because it can bypass normal user-level access boundaries. |

### Recommendation

Use Supabase Auth with one seeded email/password demo user for the first
backend implementation.

`demo_users` is an application profile table linked to `auth.users.id`. It is
not a replacement credential table. Supabase Auth remains the single source of
truth for identity and credentials; `demo_users` stores app-level display and
membership references only.

The browser must authenticate as the seeded demo Auth user and then operate
through the `authenticated` role and RLS policies. The `anon` role may access
Supabase Auth endpoints required to establish the demo session, but it must not
read or write ModelScope application tables.

The `service_role` key is server/admin-only. It may be used by seed scripts,
migrations, local setup scripts, or trusted server processes. It must never be
bundled into the frontend, checked into source control, logged to the browser,
or sent to the client under any circumstance.

This phase does not design public signup, OAuth, magic links, forgot password,
invites, account settings, or auth UI.

## Table Design

All timestamps should use `timestamptz`. All backend primary keys should be
backend-generated. Project IDs are the exception: they are stable text IDs so
the seed data can match current frontend fixtures exactly.

### `demo_users`

Application profile for the seeded demo Auth user.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key, default generated by Postgres. |
| `auth_user_id` | `uuid` | Unique reference to `auth.users.id`; identity source of truth. |
| `email` | `text` | Demo attribution email copied from Auth for display/search only. |
| `display_name` | `text` | Name shown in review history. |
| `created_at` | `timestamptz` | Insert time. |
| `updated_at` | `timestamptz` | Profile update time, admin-controlled in this slice. |

### `projects`

Durable demo project records.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `text` | Primary key. Must be one of `residential-tower-a`, `civic-center-east`, `transit-hub-02`. |
| `name` | `text` | Current fixture project name. |
| `model_label` | `text` | Current fixture model label. |
| `owner_user_id` | `uuid` | References `demo_users.id`. |
| `created_at` | `timestamptz` | Insert time. |
| `updated_at` | `timestamptz` | Update time. |

### `project_memberships`

Minimal access boundary between demo users and projects.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `text` | References `projects.id`. |
| `user_id` | `uuid` | References `demo_users.id`. |
| `role` | `text` | Constrained to `owner` or `member`. |
| `created_at` | `timestamptz` | Insert time. |

Unique constraint: `(project_id, user_id)`.

### `ai_scan_runs`

One Model Review scan attempt or seeded demo scan for a project.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `text` | References `projects.id`. |
| `created_by_user_id` | `uuid` | References `demo_users.id`. |
| `status` | `text` | `queued`, `running`, `completed`, `failed`, or `cancelled`. |
| `source` | `text` | `seed`, `demo`, or `mock`; no real AI source in this slice. |
| `started_at` | `timestamptz` | Nullable for seeded records. |
| `completed_at` | `timestamptz` | Nullable unless completed. |
| `finding_count` | `integer` | Count of associated findings. |
| `created_at` | `timestamptz` | Insert time. |

Scan completion does not create issues.

### `ai_findings`

Provisional AI review inputs produced by a scan run. An AI finding is not an
issue.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `text` | References `projects.id`. |
| `scan_run_id` | `uuid` | References `ai_scan_runs.id`. |
| `fixture_finding_id` | `text` | Current frontend fixture ID for seed alignment only. |
| `code` | `text` | Human-readable finding code such as `ISS-024`. |
| `title` | `text` | Finding title. |
| `finding_type` | `text` | `coordination`, `clearance`, `fire-safety`, or `annotation`. |
| `suggested_priority` | `text` | `critical`, `warning`, or `info`; not confirmed issue severity. |
| `confidence` | `numeric` | Nullable until frontend support exists. |
| `object_id` | `text` | Source object ID where available. |
| `object_label` | `text` | Current frontend `object` value. |
| `discipline` | `text` | `architecture`, `structure`, `mechanical`, or `electrical`. |
| `level` | `text` | Source floor/level. |
| `location` | `text` | Human-readable model location. |
| `source_payload` | `jsonb` | Structured source context needed to reconstruct frontend view models. |
| `current_status` | `text` | Optional denormalized status: `active`, `issue-created`, `dismissed`, or `follow-up`. |
| `created_at` | `timestamptz` | Insert time. |

`source_payload` should store only the minimal structured source context needed
to reconstruct `View in model` and the issue detail UI. It may include selected
fields from `ReviewIssue.details` / `ObjectDetails`, such as object ID, GUID,
category, level, elevation, material, fire rating, and lightweight geometry or
reference metadata when available. It must not store the full `ReviewIssue`
object, UI state, selected tab, preview state, rendered marker state, or other
frontend-only data. The exact payload schema should be finalized in the
migration/API implementation PR.

### `ai_finding_decisions`

Append-only user decisions on findings.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `text` | References `projects.id`. |
| `finding_id` | `uuid` | References `ai_findings.id`. |
| `scan_run_id` | `uuid` | References `ai_scan_runs.id`. |
| `user_id` | `uuid` | References `demo_users.id`. |
| `decision_type` | `text` | `create_issue`, `dismiss`, `mark_follow_up`, `restore`, or `remove_issue_link`. |
| `created_issue_id` | `uuid` | Nullable reference to `model_review_issues.id`. |
| `decision_note` | `text` | Nullable future rationale. |
| `idempotency_key` | `uuid` | Nullable key for retry-safe mutations. |
| `created_at` | `timestamptz` | Insert time. |

Rows are appended, not overwritten. `mark_follow_up` is reserved until a visible
frontend action exists. Runtime RPC functions should reject new
`mark_follow_up` writes until a future spec enables that action.

### `model_review_issues`

Durable issues created only after explicit user action.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Permanent backend issue ID. |
| `issue_code` | `text` | User-facing code generated by the backend, for example `MRI-RES-0001`. |
| `project_id` | `text` | References `projects.id`. |
| `created_by_user_id` | `uuid` | References `demo_users.id`. |
| `source_finding_id` | `uuid` | Durable source finding reference. |
| `source_scan_run_id` | `uuid` | Durable source scan run reference. |
| `source_finding_code` | `text` | Display snapshot for stable labels. |
| `title` | `text` | Issue title, initially copied from finding title. |
| `related_object` | `text` | Source object label. |
| `related_level` | `text` | Source level/floor. |
| `priority` | `text` | `critical`, `warning`, or `info`. |
| `status` | `text` | Current status. |
| `created_at` | `timestamptz` | Insert time. |
| `updated_at` | `timestamptz` | Last status/update time. |

Allowed statuses: `Open`, `In Review`, `Resolved`, `Blocked`, and
`Closed as not actionable`.

The primary key is an opaque UUID. `issue_code` is the display replacement for
browser-local `MR-001` style IDs. The recommended format is
`MRI-{PROJECT_PREFIX}-{NNNN}`, where project prefixes are seeded as `RES`,
`CIV`, and `TH`, and the numeric sequence is generated by the database per
project. The browser may create temporary optimistic IDs such as
`temp:<uuid>`, but those must be replaced by backend `id` and `issue_code` on
success.

### `issue_status_history`

Append-only status and outcome history for created issues.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `text` | References `projects.id`. |
| `issue_id` | `uuid` | References `model_review_issues.id`. |
| `from_status` | `text` | Nullable for the initial `Open` row. |
| `to_status` | `text` | New status. |
| `changed_by_user_id` | `uuid` | References `demo_users.id`. |
| `change_reason` | `text` | Nullable future rationale. |
| `changed_at` | `timestamptz` | Event time. |

Every relevant status or outcome change appends a row. Existing rows are not
updated by browser actions.

### `review_history_events`

Append-only user-facing review history for the existing Model Review workflow.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `uuid` | Primary key. |
| `project_id` | `text` | References `projects.id`. |
| `event_type` | `text` | Workflow event type. |
| `actor_user_id` | `uuid` | Nullable only for seed/system events. |
| `scan_run_id` | `uuid` | Nullable reference to `ai_scan_runs.id`. |
| `finding_id` | `uuid` | Nullable reference to `ai_findings.id`. |
| `issue_id` | `uuid` | Nullable reference to `model_review_issues.id`. |
| `decision_id` | `uuid` | Nullable reference to `ai_finding_decisions.id`. |
| `status_history_id` | `uuid` | Nullable reference to `issue_status_history.id`. |
| `label` | `text` | Existing UI-style short label. |
| `detail` | `text` | Existing UI-style event detail. |
| `created_at` | `timestamptz` | Event time. |

Persist durable workflow-changing events: scan completion, finding decisions,
issue creation, issue status changes, issue removal, and restoration. Do not
persist hover state, grouping state, panel state, preview toggles, or selected
tabs.

## RLS and Role Matrix

Use helper functions to keep policies consistent:

- `app_current_demo_user_id()` returns `demo_users.id` for `auth.uid()`.
- `app_is_project_member(project_id text)` returns true when the current demo
  user has a membership for the project.

The exact SQL belongs in the later migration PR. The design intent is:

```text
row.project_id is visible or writable only when
app_is_project_member(row.project_id) = true
```

| Table | `anon` | `authenticated` | `service_role` |
| --- | --- | --- | --- |
| `demo_users` | No table access. | SELECT own profile where `auth_user_id = auth.uid()`. No direct INSERT or UPDATE. | SELECT, INSERT, UPDATE for seeding/admin only. |
| `projects` | No table access. | SELECT projects where membership exists. No direct INSERT or UPDATE. | SELECT, INSERT, UPDATE for seeding/admin only. |
| `project_memberships` | No table access. | SELECT own membership rows. No direct INSERT or UPDATE. | SELECT, INSERT, UPDATE for seeding/admin only. |
| `ai_scan_runs` | No table access. | SELECT rows for accessible projects. No direct INSERT or UPDATE in this seeded slice. | SELECT, INSERT, UPDATE for seeding/admin only. |
| `ai_findings` | No table access. | SELECT rows for accessible projects. No direct INSERT or UPDATE. | SELECT, INSERT, UPDATE for seeding/admin only. |
| `ai_finding_decisions` | No table access. | SELECT rows for accessible projects. INSERT only through approved RPC; no direct UPDATE. | SELECT, INSERT, UPDATE for seed/admin correction only; app behavior remains append-only. |
| `model_review_issues` | No table access. | SELECT rows for accessible projects. INSERT and UPDATE only through approved RPC. | SELECT, INSERT, UPDATE for seed/admin correction only. |
| `issue_status_history` | No table access. | SELECT rows for accessible projects. INSERT only through approved RPC; no UPDATE. | SELECT, INSERT, UPDATE for seed/admin correction only; app behavior remains append-only. |
| `review_history_events` | No table access. | SELECT rows for accessible projects. INSERT only through approved RPC; no UPDATE. | SELECT, INSERT, UPDATE for seed/admin correction only; app behavior remains append-only. |

RLS must be enabled on all application tables. The `service_role` key must be
held only by trusted server/admin contexts because it is not a browser access
mechanism.

## Seed Strategy

The seed script for a later implementation should create:

1. One Supabase Auth demo user.
2. One `demo_users` profile linked to that Auth user's `auth.users.id`.
3. Three `projects` rows with IDs exactly matching current frontend fixtures:
   `residential-tower-a`, `civic-center-east`, and `transit-hub-02`.
4. One membership per project for the demo user.
5. One completed seeded scan run per project.
6. Seeded AI findings derived from the current `src/data/projects.ts`
   `projects[].issues` arrays.
7. Optional seeded decision/history rows only where needed to preserve existing
   fixture state.

The seed process should fail fast if the backend project IDs do not match the
`ProjectId` union in `src/types.ts` and the IDs exported by
`src/data/projects.ts`. Alignment should be checked by a later seed validation
script, not by changing frontend fixtures in this design phase.

Mapping from current fixtures:

| Frontend field | Backend seed target |
| --- | --- |
| `ProjectData.id` | `projects.id` |
| `ProjectData.name` | `projects.name` |
| `ProjectData.modelLabel` | `projects.model_label` |
| `ReviewIssue.id` | `ai_findings.fixture_finding_id` |
| `ReviewIssue.code` | `ai_findings.code` |
| `ReviewIssue.title` | `ai_findings.title` |
| `ReviewIssue.object` | `ai_findings.object_label` |
| `ReviewIssue.location` | `ai_findings.location` |
| `ReviewIssue.severity` | `ai_findings.suggested_priority` |
| `ReviewIssue.findingType` | `ai_findings.finding_type` |
| `ReviewIssue.discipline` | `ai_findings.discipline` |
| `ReviewIssue.details` | `ai_findings.source_payload` |

`confidence` remains nullable because current `ReviewIssue` does not expose a
confidence field. Seeded `follow-up` fixture state must not imply that runtime
`mark_follow_up` is available; the write action remains reserved until a future
frontend spec exposes it.

## RPC and Transaction Boundaries

Reads that can use Supabase client table queries with RLS:

- Resolve the current app profile from `demo_users`.
- List accessible `projects`.
- Read `ai_scan_runs`, `ai_findings`, `ai_finding_decisions`,
  `model_review_issues`, `issue_status_history`, and `review_history_events`
  for accessible projects.

Writes that must go through RPC:

- `create_issue_from_finding`
- `record_finding_decision`
- `update_issue_status`
- `remove_issue_from_tracker`, if the later implementation keeps that action

The RPC functions should be `security definer` only when needed to perform
multi-table writes while still validating the caller through `auth.uid()` and
project membership. They must not become a way around project membership.

### `create_issue_from_finding`

Inputs:

- `finding_id`
- optional `idempotency_key`
- optional display overrides that the current UI already supports

Transaction behavior:

1. Resolve current `demo_users.id` from `auth.uid()`.
2. Lock the `ai_findings` row for the supplied finding.
3. Verify project membership.
4. Verify the finding belongs to its scan run and project.
5. Reject creation if the finding already has an active created issue, or
   return the prior issue when the same idempotency key was already committed.
6. Generate `model_review_issues.id` and `issue_code`.
7. Insert `model_review_issues`.
8. Insert `ai_finding_decisions` with `decision_type = 'create_issue'`.
9. Insert initial `issue_status_history` with `from_status = null` and
   `to_status = 'Open'`.
10. Insert `review_history_events` for `issue_created`.
11. Update the optional `ai_findings.current_status` cache to `issue-created`.
12. Return the created issue, decision, status history, review history event,
   and updated finding status.

If any step fails, Postgres rolls back the whole transaction. The frontend must
not show a confirmed issue, `issue-created` finding state, status history row,
or review history event unless the RPC succeeds or a follow-up reload confirms
that an idempotent request committed.

### `record_finding_decision`

Use for append-only decisions that do not create an issue, such as `dismiss`,
`restore`, and future enabled decisions. The function should:

- verify membership
- append `ai_finding_decisions`
- update only the optional `ai_findings.current_status` cache
- append `review_history_events`
- reject `mark_follow_up` until a future frontend action is specified

It must not delete the finding or overwrite prior decisions.

### `update_issue_status`

Use for status changes. The function should:

- verify membership
- lock the issue row
- compare current status with requested status
- update `model_review_issues.status` and `updated_at`
- append `issue_status_history`
- append `review_history_events`
- return the updated issue and appended history rows

Direct table UPDATE from the browser is not allowed because it would bypass the
required append-only history behavior.

## Optimistic Frontend Failure Behavior

A later frontend integration may start optimistic actions, but backend state is
confirmed only by RPC success.

Issue creation failure:

- The UI may create a temporary pending item with a `temp:` ID.
- On success, replace it with backend `id` and `issue_code`.
- On failure, remove the temporary issue, restore the finding to the last
  confirmed backend status, show a retry path, and re-fetch project review
  state if local state is uncertain.

Network timeout after submit:

- Reuse the same `idempotency_key` when retrying.
- If the first attempt committed, the RPC returns the existing committed issue
  for that key.
- If it did not commit, the retry performs the transaction once.

Partial state:

- The database should not expose partial issue creation because the RPC is one
  transaction.
- If the browser has partial optimistic state, it must reconcile from backend
  read results.

## Frontend Integration Boundary

No runtime integration is added in this PR. A later PR may teach the existing
frontend to load and send the following data.

Frontend loads:

- current demo profile
- accessible projects
- latest project scan run state
- AI findings with current backend status
- finding decisions
- model review issues with backend IDs and issue codes
- issue status history
- review history events
- source lineage needed to reconstruct the existing issue detail view

Frontend sends:

- selected project ID
- source finding ID for issue creation or decisions
- decision type for enabled visible actions
- requested issue status transitions
- idempotency key for retryable mutations

Frontend receives:

- backend-generated issue UUID
- backend-generated display `issue_code`
- durable `source_finding_id` and `source_scan_run_id`
- appended decision row
- appended status history row
- appended review history event
- updated denormalized current finding/issue status when applicable

The existing frontend UX remains the product surface. This design does not add
upload, storage, real AI, OCR, BIM parsing, realtime collaboration, comments,
assignees, notifications, auth UI, or full issue management.

## Acceptance for Later Implementation

A later implementation PR is safe only when:

- Supabase Auth remains the identity source of truth.
- `demo_users` is only an app profile linked to `auth.users.id`.
- RLS enforces project membership for every project-scoped table.
- The browser never receives the `service_role` key.
- AI findings and issues remain separate tables.
- Scan completion never creates issue rows.
- Issue creation from finding is atomic.
- Decisions, status history, and review history are append-only.
- Browser-local IDs are temporary and backend IDs are permanent.
- Seeded project IDs match current frontend fixtures exactly.
- The existing Model Review UX remains the only product surface.
